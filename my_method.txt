================================================================================
        D-RPG: Differentiable Residual Product-Quantized Generative
                      Sequential Recommendation
================================================================================


1. INTRODUCTION
--------------------------------------------------------------------------------

This document describes D-RPG, an upgraded version of the RPG (Residual
Product-quantized Generative) sequential recommendation model. RPG belongs to
the "generative recommendation" paradigm, where each item in a catalog is
represented not by an opaque integer ID but by a compact semantic ID — a
sequence of discrete codes produced by Product Quantization (PQ) over item
content embeddings. A GPT-2 language model is then trained to autoregressively
predict the semantic ID of the next item a user will interact with, given the
sequence of items the user has interacted with so far.

The central limitation of RPG is that its semantic IDs are frozen after an
offline pre-processing step. The codebook assignments are computed once by
FAISS OPQ, cached to disk, and never revisited during model training. This
means the quantization quality — how well the discrete codes capture item
similarity — is decoupled from the recommendation objective. If the initial
quantization makes poor assignments (e.g., places dissimilar items in the same
code, or fragments similar items across codes), the downstream model has no
mechanism to correct this.

D-RPG addresses this by making the entire quantization pipeline differentiable
and trainable end-to-end. Instead of looking up pre-cached codes, D-RPG feeds
each item's sentence embedding through a Differentiable Product Quantization
(DPQ) module at every forward pass. Through the Straight-Through Estimator
(STE) trick, Gumbel-Softmax noise injection, and a self-diminishing noise
schedule, the codebooks, rotation matrix, and downstream transformer are all
optimized jointly under a single recommendation loss.


2. BACKGROUND: THE BASELINE RPG MODEL
--------------------------------------------------------------------------------

Before describing the upgrade, we review RPG in detail so that the changes are
clear.

2.1 Semantic ID Construction (Offline, Frozen)

RPG uses Optimized Product Quantization (OPQ, via FAISS) to convert each item's
sentence embedding into a compact semantic ID. The process is:

  1. Encode item metadata (title, description, etc.) into a dense vector using
     a sentence embedding model (e.g., a SentenceTransformer or OpenAI
     text-embedding-3).

  2. Optionally apply PCA with whitening to reduce dimensionality.

  3. Train a FAISS OPQ index on the training-set items. OPQ jointly learns:
     - A rotation matrix R (d × d) that aligns the embedding space so that
       each sub-vector block is decorrelated,
     - A Product Quantizer with n_digit codebooks, each of size codebook_size.

  4. Quantize every item to obtain its PQ code — a tuple of n_digit integers,
     each in [0, codebook_size). These codes are the item's semantic ID.

  5. Map semantic IDs to token IDs by offsetting each digit:
         token[k] = code[k] + k * codebook_size + 1
     This produces a flat vocabulary where digit 0 occupies tokens 1..256,
     digit 1 occupies tokens 257..512, and so on (for codebook_size=256).

  6. Cache the item→tokens mapping to disk. From this point on, semantic IDs
     are static look-up values.

The crucial point: once cached, neither the rotation matrix R nor the
codebook centroids are updated during training. The model is entirely
downstream of the frozen quantization.

2.2 Token Embedding Table

RPG stores the entire vocabulary (all digits of all codebooks, plus special
tokens) in a single nn.Embedding table — the GPT-2 word token embedding (wte).
Each digit k and codeword c has its own learnable embedding vector.

To represent an item in the input sequence, RPG looks up the item's n_digit
token IDs from the cached table, retrieves the n_digit embedding vectors from
wte, and mean-pools them into a single vector:

    input_embs = GPT2.wte(item_id2tokens[item_id]).mean(dim=-2)

This mean-pooled vector is what the GPT-2 transformer sees as the "word
embedding" for that item.

2.3 GPT-2 Transformer Backbone

The GPT-2 model processes the sequence of mean-pooled item embeddings with
causal self-attention, producing contextualized hidden states h_t for each
position t.

2.4 Prediction Heads

RPG uses n_digit independent prediction heads (one per digit). Each head is a
ResBlock (residual connection: x + SiLU(Linear(x)), zero-initialized so it
starts as identity). Each head transforms h_t into a digit-specific query:

    q_t^(k) = PredHead_k(h_t)      for k = 0, ..., n_digit - 1

2.5 Per-Digit Loss

For each digit k, RPG computes cosine similarity between the normalized query
q_t^(k) and the normalized wte embeddings of the k-th codebook:

    logits_k = (q_t^(k) / ||q_t^(k)||) · (E_k / ||E_k||)^T / temperature

where E_k is the (codebook_size × n_embd) slice of wte for digit k. A
cross-entropy loss is computed for each digit against the ground-truth code,
and the final loss is the average over all n_digit digits:

    L = (1 / n_digit) * Σ_k CE(logits_k, gt_code_k)

2.6 Generation (Inference)

At inference, RPG computes per-digit log-softmax scores and concatenates them
into a single token_logits vector. For each candidate item, it gathers the
log-probabilities of its n_digit codes and averages them:

    score(item) = (1/n_digit) Σ_k log P(code_k(item) | h_t)

The top-K items by score are returned. Optionally, a graph-constrained decoding
strategy can be used, where a k-NN item similarity graph (built from the token
embeddings) guides a local search.

2.7 Summary of RPG Limitations

  - Frozen quantization: OPQ codes are never updated during training.
  - Indirect optimization: The model learns token embeddings that must
    simultaneously serve as codebook proxies and as input features — two
    potentially conflicting roles.
  - No gradient signal to codebooks: If an item is assigned to a poor codeword,
    the only recourse is to shift the wte embeddings, not the assignment itself.


3. D-RPG: THE UPGRADED MODEL
--------------------------------------------------------------------------------

D-RPG retains the same high-level architecture (sentence embeddings → discrete
codes → GPT-2 → prediction heads → next-item scoring) but replaces the frozen
look-up with a fully differentiable quantization module and restructures the
loss to operate in the continuous DPQ embedding space.

3.1 Architectural Overview

The forward pass of D-RPG is:

  ┌─────────────────┐
  │  Item ID (int)   │
  └────────┬────────┘
           │  nn.Embedding look-up
           ▼
  ┌─────────────────┐
  │ Sentence Emb (d) │   ← stored in a learnable embedding table
  └────────┬────────┘
           │
           ▼
  ┌─────────────────────────────────────────────────────┐
  │         Differentiable Product Quantization (DPQ)    │
  │                                                      │
  │  1. Rotate:   x_rot = R(x)             [nn.Linear]  │
  │  2. Split:    x_sub = reshape to n_digit sub-vecs    │
  │  3. Assign:   logits = x_sub · K^T     [dot-product] │
  │  4. Gumbel:   noisy = logits + σ·Gumbel (train only) │
  │  5. Soft:     soft = softmax(noisy/τ) · V            │
  │  6. Hard:     hard = V[argmax(noisy)]                │
  │  7. STE:      ste = hard + soft - soft.detach()      │
  └────────┬────────────────────────────────────────────┘
           │  ste output  (n_digit * v_dim)
           │
           │  [optional input projection]
           ▼
  ┌─────────────────┐
  │    GPT-2 (h_t)   │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────────────┐
  │  n_digit Prediction Heads │  (ResBlock per digit, same as RPG)
  └────────┬────────────────┘
           │  mean over heads + output projection
           ▼
  ┌──────────────────────────────────────┐
  │  Dense retrieval:  query · items^T    │  ← items also pass through DPQ
  │  Cross-entropy loss over all items    │
  └──────────────────────────────────────┘


3.2 Sentence Embedding Table

Unlike RPG, which only stores discrete token IDs, D-RPG maintains a full
sentence embedding table as nn.Embedding:

    self.sent_emb_table = nn.Embedding.from_pretrained(sent_embs, ...)

This table is initialized from the same sentence embeddings used for OPQ
(e.g., SentenceTransformer outputs) and can optionally be fine-tuned during
training (controlled by freeze_sentence_embedding). Row 0 is a zero vector
serving as padding.

This is a fundamental difference: RPG's input path is
    item_id → cached codes → wte look-up → mean pool
while D-RPG's input path is
    item_id → sentence embedding → DPQ forward pass → STE output


3.3 Differentiable Product Quantization (DPQ) Module

The DPQ module is the core innovation. It replaces the frozen FAISS look-up
with a differentiable, trainable quantization layer. It contains three sets
of learnable parameters:

  (a) Rotation matrix R  — nn.Linear(d, d, bias=False)
      Warm-initialized from the FAISS OPQ rotation matrix.
      Purpose: align the embedding dimensions so that each sub-vector block
      captures independent information, analogous to OPQ's pre-rotation.

  (b) Key codebooks K  — nn.Parameter of shape (n_digit, codebook_size, sub_dim)
      Warm-initialized from the FAISS PQ centroids.
      Purpose: determine which codeword each sub-vector is assigned to.
      Assignment is computed as a dot product: logits = x_sub · K^T.

  (c) Value codebooks V  — nn.Parameter of shape (n_digit, codebook_size, v_dim)
      Warm-initialized from the PQ centroids (when v_dim == sub_dim).
      Purpose: provide the output representation for each codeword. Separating
      K and V allows the "lookup key" space to differ from the "output value"
      space, giving the model more flexibility (analogous to key/value
      separation in attention mechanisms).

The DPQ forward pass for an input tensor x of shape (B, seq_len, d):

  Step 1 — Rotation:
    x_rot = R(x)                                        (B, seq_len, d)

  Step 2 — Sub-vector splitting:
    x_sub = x_rot.view(B, seq_len, n_digit, sub_dim)    (B, seq_len, n_digit, sub_dim)

  Step 3 — Assignment logits (dot product with key codebooks):
    logits = einsum('bsdi, dki -> bsdk', x_sub, K)      (B, seq_len, n_digit, codebook_size)

  Step 4 — Gumbel noise injection (training only):
    gumbel = -log(-log(U))       where U ~ Uniform(0,1)
    noisy_logits = logits + σ * gumbel

  Step 5 — Soft probabilities:
    soft_probs = softmax(noisy_logits / τ, dim=-1)       (B, seq_len, n_digit, codebook_size)

  Step 6 — Soft reconstruction (weighted average over value codewords):
    soft = einsum('bsdk, dkv -> bsdv', soft_probs, V)    (B, seq_len, n_digit, v_dim)

  Step 7 — Hard reconstruction (look up the winning codeword from V):
    hard_codes = argmax(noisy_logits, dim=-1)             (B, seq_len, n_digit)
    hard = V[digit, hard_codes[digit]]                    (B, seq_len, n_digit, v_dim)

  Step 8 — Straight-Through Estimator:
    ste = hard + soft - soft.detach()                     (B, seq_len, n_digit, v_dim)

  The output is reshaped to (B, seq_len, n_digit * v_dim).


3.4 The Straight-Through Estimator (STE) Trick

The STE line deserves special attention because it is the mechanism that makes
the discrete quantization differentiable:

    ste = hard + soft - soft.detach()

Expanding what this means:

  FORWARD PASS:
    soft - soft.detach() = 0   (a tensor minus its own detached copy is zero)
    Therefore: ste = hard + 0 = hard
    → The downstream layers (GPT-2, prediction heads) receive the hard
      (discrete) reconstruction, exactly as if we did a standard VQ look-up.

  BACKWARD PASS:
    hard has no gradient (argmax is not differentiable).
    soft.detach() has no gradient (it is detached).
    soft has gradients (softmax is differentiable).
    Therefore: ∂ste/∂params = ∂soft/∂params
    → The gradient flows through the soft (continuous) reconstruction path,
      reaching the codebooks K and V, the rotation R, and the sentence
      embeddings.

This is the key insight: the model behaves discretely in the forward direction
(preserving the information bottleneck of PQ), but trains continuously in the
backward direction (allowing gradient-based optimization of all quantization
parameters).


3.5 Gumbel Noise for Exploration

During training, Gumbel noise is added to the assignment logits:

    gumbel = -log(-log(U)),   U ~ Uniform(0,1)
    noisy_logits = logits + σ * gumbel

The Gumbel distribution is the standard noise distribution for the
Gumbel-Softmax / Gumbel-Max trick. It has the property that
argmax(logits + Gumbel) samples from the categorical distribution defined by
softmax(logits), rather than always picking the mode.

Why this matters:

  (a) Exploration: Without noise, the assignment would always go to the
      highest-scoring codeword. Gumbel noise introduces stochasticity,
      allowing the model to occasionally try alternative codewords and
      discover better assignments.

  (b) Codebook collapse prevention: A well-known failure mode of VQ is
      "codebook collapse," where only a small fraction of codewords are ever
      used and the rest become dead. By randomly perturbing assignments,
      Gumbel noise gives underutilized codewords a chance to be selected,
      receive gradient updates, and become useful.

  (c) Consistent sampling: The Gumbel-Max trick ensures that the stochastic
      assignments are properly calibrated — the probability of selecting
      codeword c is proportional to exp(logit_c), not uniform.

During inference, no Gumbel noise is added (σ = 0 effectively), and the
model uses pure argmax for deterministic predictions.


3.6 Noise Annealing: Self-Diminishing Uncertainty-Driven (SDUD) Schedule

D-RPG does not use a fixed noise scale σ. Instead, it adapts σ based on
the model's current training loss using a self-diminishing schedule:

    running_loss ← 0.99 * running_loss + 0.01 * current_batch_loss  (EMA)
    σ ← max(0,  √(running_loss) - λ)

where λ (sdud_lambda) is a threshold hyperparameter (default: 1.4).

The intuition is:

  - Early in training, the loss is high (the model is uncertain), so
    √(loss) > λ and σ is positive → strong Gumbel noise encourages
    exploration.

  - As training progresses and the loss drops, √(loss) approaches λ,
    and σ shrinks toward zero → the model transitions from exploration
    to exploitation, settling on stable assignments.

  - Once √(loss) < λ, σ = 0 and the model is in pure exploitation mode
    with no stochasticity in assignments.

This is adaptive and requires no manual noise schedule: the model
automatically determines when it has learned enough to stop exploring.

Additionally, D-RPG anneals the Gumbel-Softmax temperature τ via exponential
decay at the end of each training epoch:

    τ ← max(τ_min, τ * τ_decay)

Lower τ makes the soft probabilities sharper (closer to one-hot), which makes
the soft reconstruction closer to the hard reconstruction, reducing the
STE approximation error. By the end of training, τ is near τ_min and the
soft path closely mirrors the hard path.


3.7 Warm Initialization from FAISS

A naive random initialization of the DPQ parameters would discard all the
structure learned by FAISS OPQ and force the model to rediscover it from
scratch. D-RPG avoids this by warm-initializing from the FAISS index:

  - Rotation R: initialized from the OPQ rotation matrix (the linear
    transform that FAISS learned to decorrelate sub-vectors).
  - Key codebooks K: initialized from the PQ centroids.
  - Value codebooks V: initialized from the PQ centroids (when v_dim matches
    sub_dim).

This means D-RPG starts from exactly the same quantization as RPG and can
only improve from there. The tokenizer (D_RPGTokenizer) extends RPGTokenizer
to additionally:
  1. Store the full sentence embeddings (not just the codes),
  2. Save the FAISS index to disk (not just the codes),
  3. Extract the rotation matrix and PQ centroids from the saved index.


3.8 Loss Function: Dense Retrieval Instead of Per-Digit Classification

RPG computes a separate cross-entropy loss for each of the n_digit digits:

    L_RPG = (1/n_digit) Σ_k CE(q_k · E_k^T / τ,  gt_code_k)

where q_k is the k-th head's output and E_k is the k-th codebook's wte slice.
This decomposes the problem into n_digit independent classification tasks.

D-RPG replaces this with a single dense-retrieval cross-entropy:

    query = normalize(OutputProj(mean_k(PredHead_k(h_t))))
    item_embs = normalize(DPQ(sent_emb_table[1:]))       ← all items
    logits = query · item_embs^T / τ
    L_D-RPG = CE(logits, gt_item_id - 1)

Key differences:

  (a) Unified scoring: Instead of scoring each digit independently, D-RPG
      scores whole items at once. The query is the mean of all digit-specific
      heads, projected into the DPQ embedding space. This allows cross-digit
      interactions.

  (b) Items pass through DPQ: The candidate item embeddings are not static
      look-ups — they are produced by running the sentence embeddings through
      the same DPQ module (with STE). This means the item representations
      are also trainable end-to-end.

  (c) Gradient to DPQ via both paths: The DPQ module receives gradients from
      two sources:
      - The INPUT path: changing DPQ parameters changes the input to GPT-2.
      - The SCORING path: changing DPQ parameters changes the item embeddings
        used for retrieval scoring.

  (d) Softmax over all items: The denominator of the cross-entropy includes
      all items in the catalog (not just the codewords of one digit), giving
      a more globally calibrated probability distribution.


3.9 Generation (Inference) Changes

At inference, D-RPG also differs from RPG:

  RPG:
    - Computes per-digit log-softmax scores from token embeddings.
    - For each candidate item, gathers the n_digit log-probabilities of
      its cached codes and averages them.

  D-RPG:
    - Computes a single query vector by mean-pooling the n_digit head outputs
      and projecting to DPQ space.
    - Computes hard DPQ embeddings for all items via _get_all_item_embs().
    - Scores items by cosine similarity: query · item_embs^T / τ.
    - Returns top-K by score.

The graph-constrained decoding is also adapted: in RPG the graph is built from
per-digit token embedding similarity; in D-RPG it is built from DPQ hard
embedding similarity. D-RPG's graph_propagation receives item_logits (scores
over items) directly, rather than token_logits (scores over per-digit codes).


3.10 Optional Projection Layers

If the DPQ output dimension (n_digit * v_dim) differs from the GPT-2 hidden
dimension (n_embd), D-RPG inserts linear projection layers:

  - input_proj:  DPQ output → GPT-2 input   (before the transformer)
  - output_proj: GPT-2 output → DPQ space   (after the prediction heads)

When the dimensions match (the default configuration where v_dim = n_embd /
n_digit), these are identity mappings.

5. DESIGN RATIONALE
--------------------------------------------------------------------------------

5.1 Why pass through the full quantization pipeline instead of using cached IDs?

In RPG, the model only ever sees the discrete codes — it has no access to the
original continuous embedding and no way to know whether the quantization was
good or bad. By re-running quantization at every forward pass, D-RPG gives the
optimizer a lever to improve the codes themselves. If the loss says "this item
was poorly scored," the gradient can flow back through the STE, through the
soft probabilities, and into the codebooks and rotation, pushing the
quantization toward assignments that are more useful for recommendation.

5.2 Why STE rather than pure Gumbel-Softmax relaxation?

A pure Gumbel-Softmax (using only soft output) would train well but would
create a train-test mismatch: during training the model sees soft (continuous)
reconstructions, but at test time it must use hard (discrete) codes. The STE
ensures that the forward pass always uses hard codes, so the model trains
under the same discrete bottleneck it will face at inference. The soft path
is used only to supply gradients.

5.3 Why separate Key (K) and Value (V) codebooks?

In standard PQ, the same centroids serve as both the "matching key" (which
codeword is closest?) and the "output value" (what vector do we output?).
By decoupling K and V, D-RPG allows the assignment criterion to differ from
the output representation. The keys can specialize in discriminating sub-vector
clusters, while the values can specialize in producing features useful for the
downstream GPT-2 model. This is analogous to the key-value separation in
attention mechanisms.

5.4 Why Gumbel noise specifically?

Gumbel noise, combined with argmax, yields an exact sample from the categorical
distribution softmax(logits). Other noise distributions (e.g., Gaussian) would
not have this property and would bias the exploration. The Gumbel-Max trick
ensures that the exploration is calibrated: codewords with higher logits are
explored more often, but all codewords have a non-zero chance.

5.5 Why SDUD noise annealing?

A fixed noise schedule (e.g., linear decay over N epochs) requires manual
tuning and cannot adapt to different datasets or training dynamics. The SDUD
schedule (σ = max(0, √EMA_loss - λ)) is self-adaptive: it uses the model's
own loss as a signal for how much exploration is still needed. When the model
is confused (high loss), it explores more; when the model is confident (low
loss), it exploits. The only hyperparameter is λ, which sets the loss
threshold below which exploration stops entirely.


6. TECHNICAL DETAILS
--------------------------------------------------------------------------------

6.1 DPQ Hyperparameters

  - d (int): Sentence embedding dimension (from the pre-trained encoder).
  - n_digits (int): Number of PQ sub-spaces (= n_codebook in config).
  - codebook_size (int): Number of codewords per sub-space.
  - v_dim (int): Dimension of each value codeword (default: n_embd / n_digits).
  - sub_dim (int): d / n_digits, the dimension of each key sub-vector.

6.2 Temperature and Noise Parameters

  - quantizer_temperature (τ): Initial Gumbel-Softmax temperature (default: 1.0).
  - min_quantizer_temperature (τ_min): Floor for τ (default: 0.1).
  - quantizer_temperature_decay: Multiplicative decay per epoch (default: 0.9).
  - sdud_lambda (λ): SDUD threshold (default: 1.4).
  - σ: Gumbel noise scale, adaptively computed from the EMA of training loss.

6.3 Training Loop Integration

  At the end of each training epoch, the model's on_train_epoch_end() method:
    1. Calls anneal_tau() to decay τ.
    2. Logs the current τ and σ values for monitoring.

  During each forward pass (when training):
    1. The DPQ module receives the current τ and σ.
    2. Gumbel noise is sampled and scaled by σ.
    3. Soft probabilities are computed with temperature τ.
    4. The training loss EMA is updated, and σ is recomputed via SDUD.

6.4 D_RPGTokenizer Extensions

  The D_RPGTokenizer extends RPGTokenizer with three additional attributes:

    - sent_embs (np.ndarray, shape (n_items, d)):
      Full sentence embeddings, padded with a zero row at index 0. Used to
      initialize the model's sent_emb_table.

    - opq_rotation (np.ndarray, shape (d, d) or None):
      The OPQ rotation matrix extracted from the FAISS index. Used to
      warm-initialize DPQ's rotation layer.

    - pq_codebooks (np.ndarray, shape (n_digits, codebook_size, sub_dim) or None):
      The PQ centroids extracted from the FAISS index. Used to warm-initialize
      DPQ's K and V codebooks.

  The tokenizer saves the FAISS index to disk (in addition to the semantic IDs)
  so that these parameters can be extracted even if the index is rebuilt.


7. CONCLUSION
--------------------------------------------------------------------------------

D-RPG upgrades the RPG model from a "frozen quantization + downstream
transformer" pipeline to a fully end-to-end differentiable system. The key
contribution is the DPQ module, which replaces static FAISS look-ups with a
trainable quantization layer that uses:

  1. The Straight-Through Estimator to maintain discrete codes in the forward
     pass while allowing gradient flow in the backward pass.

  2. Gumbel noise injection to encourage exploration and prevent codebook
     collapse.

  3. A self-diminishing noise schedule (SDUD) that automatically transitions
     from exploration to exploitation as training progresses.

  4. Warm initialization from the FAISS OPQ parameters to start from a strong
     quantization baseline rather than random.

  5. A unified dense retrieval loss that replaces per-digit classification with
     full-item scoring, enabling cross-digit interactions and end-to-end
     gradient flow through both the input and scoring paths.

These changes allow the quantization — the way items are represented as
discrete codes — to be jointly optimized with the recommendation objective,
closing the gap between the "how items are encoded" and "how items are
recommended" stages that exists in the baseline RPG model.
